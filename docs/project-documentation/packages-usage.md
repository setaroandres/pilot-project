## `src/lib/prisma.ts`

```ts
import { PrismaClient } from "@/generated/prisma/client";
import { PrismaPg } from "@prisma/adapter-pg";
import { createPrismaClient } from "@upstart13-com/aiden-db";

const adapter = new PrismaPg({
  connectionString: process.env.DATABASE_URL!,
});

export const prisma = createPrismaClient(PrismaClient, { adapter });
```

### What it is

The single Prisma client for the entire app. It takes the Prisma 7 consumer-generated client (`@/generated/prisma/client`, gitignored, regenerated by `npm run prisma:generate`), attaches the `pg` driver adapter (`PrismaPg`), and wraps both through `createPrismaClient` from `@upstart13-com/aiden-db` — which layers the SDK's connection handling/instrumentation on top of a plain `PrismaClient`. Every other file in the app imports `{ prisma }` from `@/lib/prisma`; nothing constructs its own client.

Server-only: it must never be imported into a `"use client"` file, since the generated Prisma client can't run in the browser.

### What it's used for

All database access — reads, writes, updates, and deletes. There is no other path to Postgres in this app.

- **Read**: `prisma.catalogEntry.findMany({ where: {...} })` — `src/app/api/catalog/route.ts`; `prisma.user.findMany(...)` / `prisma.role.findMany(...)` — `src/app/admin/users/page.tsx`.
- **Read single**: `prisma.conversation.findUnique({ where: { id } })` — `src/app/api/conversations/[id]/route.ts`.
- **Create**: `createRegisterHandler({ prisma })` inserts a new `User` row on registration; the AI usage sink in `lib/ai.ts` does `prisma.aIUsage.create({ data })` on every AI call; `createPrismaAuditSink` writes rows to `audit_logs`.
- **Delete**: `prisma.pinnedVisualization.delete({ where: { id } })` — `src/app/api/pins/[id]/route.ts`.
- **Update**: role assignment in `src/app/api/admin/users/[id]/roles/route.ts`.

### Where it's imported

**API routes:** `api/admin/cost`, `api/admin/users`, `api/admin/users/[id]/roles`, `api/auth/register`, `api/catalog`, `api/catalog/[id]`, `api/conversations`, `api/conversations/[id]`, `api/conversations/[id]/turns`, `api/me`, `api/me/export`, `api/me/password`, `api/pins`, `api/pins/[id]`.

**Server component pages:** `admin/cost`, `admin/users`, `dashboard/catalog`, `dashboard/conversations`, `dashboard/conversations/[id]`, `dashboard/pins`, `dashboard/settings/profile`, `dashboard/settings/security`.

**Other `lib/` wiring modules:** `lib/auth.ts` (passed into `createAuth`/`credentialsProvider`), `lib/audit.ts` (feeds `createPrismaAuditSink`), `lib/ai.ts` (AI usage sink), `lib/query-engine.ts` (executes AI-generated queries).

### Security boundary worth noting

The AI-driven query feature (`query-engine.ts`) never lets the AI author raw SQL against `prisma`. The model only returns a structured `QuerySpec`, which the app translates into a safe, parameterized Prisma call. No `$executeRawUnsafe`, no AI-authored SQL touching the database — consistent with this project's security rules.

## `src/lib/auth.ts`

```ts
import {
  createAuth, credentialsProvider, githubProvider, googleProvider, microsoftEntraIdProvider,
} from "@upstart13-com/aiden-auth";
import { prisma } from "@/lib/prisma";
import { aidenConfig } from "@/../aiden.config";

const providers = [];
// ...providers pushed conditionally based on aidenConfig.auth.providers.*

export const { handlers, signIn, signOut, auth } = createAuth({ prisma, providers });
```

### What it is

Configures NextAuth via `createAuth` from `@upstart13-com/aiden-auth`. Providers (Google, GitHub, Microsoft Entra ID, credentials) are registered conditionally based on flags in `aiden.config.ts`. Exports four things: `handlers` (raw NextAuth route handlers), `signIn`/`signOut` (server-side helpers), and `auth()` (reads the current session).

### What it's used for

**Route wiring** — `src/app/api/auth/[...nextauth]/route.ts` forwards every `/api/auth/*` request to NextAuth's internals:
```ts
import { handlers } from "@/lib/auth";
export const { GET, POST } = handlers;
```
This single catch-all route (Next.js dynamic + catch-all segment syntax) handles sign-in, OAuth callbacks, session checks, and sign-out — none of these have dedicated app routes except registration.

**Session gating in server components** — nearly every page/layout under `dashboard/`, `admin/`, and `(marketing)/` calls `auth()` directly to read the session and redirect unauthenticated users, e.g. `src/app/dashboard/page.tsx`:
```ts
const session = await auth();
if (!session?.user?.id) redirect("/login?callbackUrl=/dashboard");
```

**Login** has no dedicated API route. `<LoginForm />` (from `aiden-auth/components`, rendered in `src/app/(auth)/login/page.tsx`) calls `signIn("credentials", {...})` client-side (from `next-auth/react`), which POSTs to `/api/auth/callback/credentials` — swallowed by the `[...nextauth]` catch-all.

**Register** does have a dedicated route, `src/app/api/auth/register/route.ts`, because it needs custom app logic (IP rate limiting via `withRateLimit`, a `confirmPassword` normalization shim) on top of `createRegisterHandler({ prisma })`.

**Logout** — `signOut({ callbackUrl: "/" })` is called directly from `next-auth/react` in `src/components/settings/security-section.tsx`'s "Sign out" button (client-side, not via `@/lib/auth` since that export is server-only).

`security.ts` also imports `auth` from this module to wire it into `configureSecurity`.

## `src/lib/security.ts`

```ts
import { configureSecurity } from "@upstart13-com/aiden-security";
import { auth } from "@/lib/auth";
import "@/lib/audit"; // side effect: registers the Prisma audit sink

configureSecurity({ getSession: () => auth() });

export {
  withAuth, parseRequest, assertOwnership, assertCan, auditLog,
  RequestValidationError, OwnershipError, AbilityError,
} from "@upstart13-com/aiden-security";
```

### What it is

Calls `configureSecurity` once (telling `aiden-security` how to fetch the current session), then re-exports the package's core primitives. This is the only file every API route imports from for auth/validation/authorization.

### What it's used for

Every route under `src/app/api/**/route.ts` follows the same shape:

```ts
export const GET = withAuth(async (req, { session, params }) => {
  // session is guaranteed non-null here; withAuth returns 401 otherwise
});
```

- **`withAuth`** — wraps every handler. Decodes the session, injects `{ session, params, requestId }` into the handler, returns 401 if unauthenticated, and catches three known error types thrown inside the handler (`RequestValidationError` → 400, `OwnershipError` → 404, `AbilityError` → 403, with an automatic audit log on the 403 case).

  **Important nuance: `withAuth` only handles authentication, not authorization.** It answers "is someone logged in?" — not "is this logged-in user allowed to do this?" If there's no session it returns 401 before the handler body runs; if there is a session, it hands over `{ session, params, requestId }` and the handler proceeds regardless of role or ownership. Authorization is a separate, opt-in step written inside the handler:

  - `assertCan(abilities, session, "action")` — role/permission check (403 if the role doesn't match)
  - `assertOwnership(row, session.user.id)` — row-level "is this yours" check (404 if not)

  Some routes only need `withAuth` and nothing else — e.g. `api/catalog/route.ts` has no `assertCan`/`assertOwnership` call at all, because "any logged-in user can browse the catalog" is the intended rule. Others stack `withAuth` + `assertCan` (`api/admin/audit/route.ts`), `withAuth` + `assertOwnership` (`api/conversations/[id]/route.ts`), or all three together (`api/conversations/[id]/turns/route.ts`).

  **Scope: API routes only.** `withAuth` wraps `route.ts` handlers under `src/app/api/**`. Pages/layouts (`page.tsx`, `layout.tsx`) never use it — they call `auth()` directly from `@/lib/auth` and manually `redirect()` if there's no session (see `dashboard/page.tsx`, `admin/users/page.tsx` in the `auth.ts` section above). Same underlying session, two different guard mechanisms: `withAuth` returns a 401 JSON response for API endpoints, `auth()` + `redirect()` sends a rendered page to `/login`.

  One-line takeaway: `withAuth` guarantees a route only runs for authenticated users; whether that user is *authorized* for the specific action is still something the route author must check explicitly with `assertCan`/`assertOwnership` inside the handler.

- **`parseRequest(req, zodSchema)`** — validates request bodies with Zod. Used in mutation routes, e.g. `api/pins/route.ts`, `api/me/route.ts`, `api/admin/users/[id]/roles/route.ts`.

  **Direction matters: it validates the incoming request body (client → server), not a database query result.** Its implementation is a thin wrapper: read `req.json()`, run `schema.safeParse(json)`, throw `RequestValidationError` on failure (which `withAuth` turns into a 400 with the Zod error details), otherwise return the typed, validated data. It always runs *before* any Prisma call — the gate that decides whether an incoming payload is well-formed enough to act on. Zod never validates what comes back *from* the database; Prisma's own generated types cover that side. Example, `src/app/api/pins/route.ts`:
  ```ts
  const Body = z.object({
    conversationTurnId: z.string().cuid(),
    title:              z.string().min(1).max(200),
    resultSnapshot:     z.array(z.record(z.string(), z.unknown())).optional(),
  });

  export const POST = withAuth(async (req, { session }) => {
    assertCan(abilities, session, "query.run");
    const { conversationTurnId, title, resultSnapshot } = await parseRequest(req, Body);
    // only now, with validated data, do we touch prisma
    const turn = await prisma.conversationTurn.findUnique({ where: { id: conversationTurnId } });
    // ...
  });
  ```

- **`assertOwnership(row, userId)`** — the IDOR guard. Throws `OwnershipError` (→ 404) if a fetched row doesn't belong to the caller. Example, `src/app/api/conversations/[id]/route.ts`:
  ```ts
  const conversation = await prisma.conversation.findUnique({ where: { id }, include: { turns: {...} } });
  assertOwnership(conversation, session.user.id);
  ```
  And `src/app/api/pins/[id]/route.ts` combines it with `auditLog`:
  ```ts
  const pin = await prisma.pinnedVisualization.findUnique({ where: { id } });
  assertOwnership(pin, session.user.id);
  await prisma.pinnedVisualization.delete({ where: { id } });
  auditLog({ event: "pin.delete", actorId: session.user.id, resourceId: id });
  ```

- **`assertCan(abilities, session, action)`** — role/permission gate. See the `abilities.ts` section below.

- **`auditLog(...)`** — writes an entry via the Prisma audit sink registered by `lib/audit.ts`.

## `src/lib/abilities.ts`

```ts
import { defineAbilities } from "@upstart13-com/aiden-security";
import { ROLES } from "@/config/rbac";

const ADMIN = ROLES.find((r) => r.name === "admin")!.name;
const ANALYST = ROLES.find((r) => r.name === "analyst")!.name;

export const abilities = defineAbilities({
  rules: {
    "audit.read":     { roles: [ADMIN] },
    "audit.export":   { roles: [ADMIN] },
    "users.manage":   { roles: [ADMIN] },
    "query.run":      { roles: [ADMIN, ANALYST] },
    "catalog.manage": { roles: [ADMIN] },
    "cost.view":      { roles: [ADMIN] },
  },
});
```

### What it is

The app-wide RBAC rule map, built once via `defineAbilities` from `aiden-security`. It maps action strings (which double as the permission keys seeded into the database from `src/config/rbac.ts` — the single source of truth for role/permission names) to either `{ roles: [...] }` shorthand or a predicate `(session, resource) => boolean` for resource-level checks. Multiple rules on one action are OR'd. `defineAbilities` returns an object with two methods: `can(session, action, resource?)` (boolean check) and `assert(session, action, resource?)` (throws `AbilityError` if `can` returns false).

### What it's used for

**In API routes**, via the `assertCan` free function re-exported from `security.ts` — e.g. `src/app/api/admin/audit/route.ts`:
```ts
export const GET = withAuth(async (req, { session }) => {
  assertCan(abilities, session, "audit.read");
  // ...
});
```
`assertCan(abilities, session, action)` is a one-line pass-through to `abilities.assert(session, action)` — functionally identical, offered as a flat function purely for API-surface consistency with the rest of `aiden-security`'s exports (`withAuth`, `parseRequest`, `assertOwnership` are all flat functions too). `abilities` has to be passed explicitly because it isn't a singleton the package can reach into on its own — it's a plain object this app constructed, and the package has zero built-in knowledge of what `"audit.read"` means or which roles exist.

If the check fails, `AbilityError` propagates to `withAuth`'s catch block, which automatically audit-logs the denial (`event: "security.ability_denied"`) and returns a 403 — no manual error handling needed in the route.

**In server components**, for UI-level gating (not just data protection) — `src/app/admin/users/page.tsx`:
```ts
if (!abilities.can(session as never, "users.manage")) {
  redirect("/dashboard");
}
```
Here `.can()` is called directly (no throw needed) since the response is a redirect rather than a JSON error.

### Two-layer model

Authentication (`withAuth`, "are you signed in?") and authorization (`assertCan`, "are you allowed to do *this*?") are independent layers. Routes that are safe for any signed-in user (e.g. `GET /api/catalog`) skip `assertCan` entirely — `withAuth` alone is sufficient. See `docs/project-documentation/rbac.md` for the full roles/permissions matrix.

## `src/lib/ai.ts` and `src/lib/ai-mock.ts`

`@upstart13-com/aiden-ai` is the provider-agnostic AI client — one interface (`AIClient` with `.complete()` and `.stream()`) across OpenAI, Anthropic, Google, Mistral, Groq, and Cohere, with built-in retry/backoff and automatic cost/token/latency telemetry via `@upstart13-com/aiden-logging`. This app wraps it through two `lib/` files.

### `src/lib/ai.ts` — lazy provider factories

```ts
import { createAIClient, type AIClient } from "@upstart13-com/aiden-ai";
import { setAIUsageSink }                from "@upstart13-com/aiden-logging";
import { MockAIClient }                  from "@/lib/ai-mock";

setAIUsageSink(async (record) => {
  const { requestId: _requestId, ...data } = record;
  await prisma.aIUsage.create({ data });
});

export const ai = {
  mock: (): Promise<AIClient> => Promise.resolve(new MockAIClient()),
  openai: (model = providerModels.openai.model ?? "gpt-4o-mini") =>
    createAIClient({ provider: "openai", model, apiKey: process.env.OPENAI_API_KEY }),
  anthropic: (model = providerModels.anthropic.model ?? "claude-haiku-4-5") =>
    createAIClient({ provider: "anthropic", model, apiKey: process.env.ANTHROPIC_API_KEY }),
  // ...google, mistral, groq, cohere follow the same pattern
};
```

Each entry in `ai` is a *function*, not a pre-built client, so calling `ai.openai()` is what actually invokes `createAIClient` — a provider you never call is never touched, and its SDK never needs to be installed (each provider SDK is an optional peer dependency of `aiden-ai`).

**Side effect on import:** this module calls `setAIUsageSink` once, registering a sink that writes one row to the `ai_usage` table (`prisma.aIUsage.create`) for **every** AI call in the app — mock or real, `.complete()` or `.stream()`. No route has to remember to log usage; it's baked into the client layer the moment `@/lib/ai` is imported.

### `src/lib/ai-mock.ts` — `MockAIClient`

The app's own zero-cost implementation of the `AIClient` interface, satisfying a "mock provider" requirement for this pilot without calling any real API. `buildMockText()` returns deterministic, keyword-driven text — or structured JSON matching the query engine's `QuerySpec` shape when a `responseSchema` is requested — based on words in the last user message (e.g. "revenue" → a financial-records mock spec, "readmission" → patient-outcomes). It still calls `reportUsage()` (from `aiden-ai`) so the telemetry pipeline behaves identically to a real provider, just with synthetic token counts and `costUSD: 0`. Because it implements the same `AIClient` interface as real providers, callers can swap between `ai.mock()` and `ai.anthropic()` etc. with no other code changes.

### Where it's called

**1. `src/lib/query-engine.ts`** — the core NL-to-analytics engine. Picks the client with a single config flag:
```ts
const useMock = aidenConfig.ai.providers.mock.enabled;
const client  = useMock ? await ai.mock() : await ai.anthropic();

const response = await client.complete({
  system: systemPrompt,
  messages,
  responseSchema: QUERY_SPEC_JSON_SCHEMA, // derived from the same Zod schema used to validate the result
  temperature: 0,
});
```
It forces structured output so the AI can only return a `QuerySpec` descriptor — never raw SQL — then re-validates `response.parsed` with `QuerySpecSchema.safeParse` before trusting it, even though `responseSchema` was already requested. The validated spec is what becomes a parameterized `prisma.$queryRaw` call, with every column name checked against a per-entity allowlist (see the `prisma.ts` section above for the full security rationale).

**2. `src/app/api/ai/chat/route.ts`** — a free-form streamed chat endpoint. Picks whichever provider has an API key set:
```ts
async function firstAvailableProvider() {
  if (process.env.OPENAI_API_KEY) return ai.openai();
  if (process.env.ANTHROPIC_API_KEY) return ai.anthropic();
  // ...falls through google, mistral, groq, cohere
}
```
The response streams straight back over SSE via `createAIStreamResponse` (from `@upstart13-com/aiden-realtime`). Wrapped in `withAuth` + `withRateLimit` (20 requests/min per `session.user.id`) specifically to cap provider spend per account.

**3. `src/app/api/narrate/route.ts`** — always calls `ai.mock()` explicitly (not provider-switchable) to generate a 2–3 sentence narration of query results, streamed the same way, then calls `auditLog({ event: "ai.narrate", ... })` so the full query → narration lifecycle is auditable.

### Appendix — implementing an `AIClient` from scratch (mock and real, step by step)

This is a from-first-principles build-up of the interface above, useful if you're adding a new provider or want to understand exactly what `MockAIClient` and `createAIClient` are doing under the hood.

#### The contract

Every provider — real or fake — implements the same interface:

```ts
interface AIClient {
  readonly provider: string;
  readonly model: string;
  complete(options: AICompleteOptions): Promise<AICompleteResponse>;
  stream(options: AICompleteOptions): Promise<AIStreamResponse>;
}
```

```ts
interface AICompleteOptions {
  messages: AIMessage[];               // { role: "system"|"user"|"assistant"|"tool", content: string }
  system?: string;
  temperature?: number;
  maxTokens?: number;
  tools?: AIToolDefinition[];          // tool-use / function calling
  responseSchema?: Record<string, unknown>; // request structured output
  signal?: AbortSignal;
}

interface AICompleteResponse {
  text: string;                        // empty string if only tool calls came back
  toolCalls: AIToolCall[];
  parsed?: unknown;                    // populated only if responseSchema was passed
  usage: AIUsage;                      // { promptTokens, completionTokens, totalTokens, costUSD, latencyMs }
  finishReason: string;
  model: string;
}
```

`stream()` returns the same shape delivered as `{ delta: string }` chunks, plus a `finalResponse()` awaitable once the stream ends. Because both a mock and a real client satisfy this exact interface, calling code never needs to know which one it's talking to.

#### Step 1 — Install


#### Step 2 — Register the usage sink once, before either client is used

```ts
import { setAIUsageSink } from "@upstart13-com/aiden-logging";
import { prisma } from "@/lib/prisma";

setAIUsageSink(async (record) => {
  await prisma.aIUsage.create({ data: record });
});
```

Every `complete()`/`stream()` call from any client — mock or real — fires this sink automatically with real or synthetic `{ provider, model, promptTokens, completionTokens, totalTokens, costUSD, latencyMs }`. It's registered once, not per call.

#### Step 3 — Build the mock client

```ts
import type { AIClient, AICompleteOptions, AICompleteResponse, AIStreamResponse } from "@upstart13-com/aiden-ai";
import { reportUsage } from "@upstart13-com/aiden-ai";

function buildMockText(options: AICompleteOptions): string {
  const lastUser = [...options.messages].reverse().find((m) => m.role === "user")?.content ?? "";
  if (lastUser.toLowerCase().includes("weather")) return "It's sunny and 72°F.";
  return "Mock AI provider active. No real API call was made.";
}

export class MockAIClient implements AIClient {
  readonly provider = "mock";
  readonly model = "mock-1.0";

  async complete(options: AICompleteOptions): Promise<AICompleteResponse> {
    const start = Date.now();
    const text = buildMockText(options);

    const usage = reportUsage({
      provider: this.provider,
      model: this.model,
      promptTokens: 420,       // synthetic but stable
      completionTokens: 180,
      latencyMs: Date.now() - start,
    });

    let parsed: unknown;
    if (options.responseSchema) {
      try { parsed = JSON.parse(text); } catch { /* leave undefined on malformed JSON */ }
    }

    return { text, toolCalls: [], parsed, usage, finishReason: "stop", model: this.model };
  }

  async stream(options: AICompleteOptions): Promise<AIStreamResponse> {
    const start = Date.now();
    const text = buildMockText(options);
    const parts = text.split(/(?<=\s)|(?=\s)/).filter((p) => p.length > 0);
    let cached: AICompleteResponse | undefined;

    return {
      [Symbol.asyncIterator]() {
        let i = 0;
        return {
          async next() {
            if (i >= parts.length) return { done: true, value: undefined as any };
            await new Promise((r) => setTimeout(r, 10)); // simulated network latency
            return { done: false, value: { delta: parts[i++] } };
          },
        };
      },
      async finalResponse() {
        if (cached) return cached; // safe to call more than once
        const usage = reportUsage({ provider: "mock", model: "mock-1.0", promptTokens: 420, completionTokens: 180, latencyMs: Date.now() - start });
        cached = { text, toolCalls: [], parsed: undefined, usage, finishReason: "stop", model: "mock-1.0" };
        return cached;
      },
    };
  }
}
```

That's a complete, working mock — no API key, `costUSD` always 0, and it still emits realistic telemetry through the sink registered in Step 2.

#### Step 4 — Build a real client

No adapter to write yourself — `createAIClient` lazily imports the matching provider SDK:

```ts
import { createAIClient } from "@upstart13-com/aiden-ai";

const client = await createAIClient({
  provider: "anthropic",
  model: "claude-haiku-4-5",
  apiKey: process.env.ANTHROPIC_API_KEY,
});
```

Same call shape as the mock:

```ts
const response = await client.complete({
  system: "You are a concise assistant.",
  messages: [{ role: "user", content: "Summarize Fermat's last theorem." }],
  temperature: 0,
});
console.log(response.text);
console.log(response.usage); // real token counts, real costUSD
```

Streaming, structured output, and tool use all use the identical method signatures as the mock:

```ts
// Streaming
const stream = await client.stream({ messages: [{ role: "user", content: "Write a haiku about caching." }] });
for await (const chunk of stream) process.stdout.write(chunk.delta);
const final = await stream.finalResponse();

// Structured output
const structured = await client.complete({
  messages: [{ role: "user", content: "Extract: name=Ada, age=37" }],
  responseSchema: { type: "object", properties: { name: { type: "string" }, age: { type: "number" } }, required: ["name", "age"] },
});
console.log(structured.parsed); // { name: "Ada", age: 37 }

// Tool use
const toolResponse = await client.complete({
  messages: [{ role: "user", content: "What's the weather in Tokyo?" }],
  tools: [{ name: "get_weather", description: "Look up current weather", parameters: { type: "object", properties: { city: { type: "string" } }, required: ["city"] } }],
});
for (const call of toolResponse.toolCalls) console.log(call.name, JSON.parse(call.arguments));
```

Optional retry tuning (defaults: 3 attempts, exponential backoff from 250ms, retried on 429/5xx/network errors):

```ts
const client = await createAIClient({
  provider: "openai",
  model: "gpt-4o-mini",
  apiKey: process.env.OPENAI_API_KEY,
  retry: { attempts: 5, initialDelayMs: 500 },
});
```

#### Step 5 — Wire both into one switchable factory

```ts
export const ai = {
  mock: () => Promise.resolve(new MockAIClient()),
  anthropic: (model = "claude-haiku-4-5") =>
```

#### Step 6 — Full route example, mock vs. real

Mock (no API key needed, safe for local dev/CI):

```ts
export const POST = withAuth(async (req) => {
  const { prompt } = await parseRequest(req, z.object({ prompt: z.string().min(1) }));
  const client = await ai.mock();
  const response = await client.complete({ messages: [{ role: "user", content: prompt }] });

```

Real provider (identical structure; add rate limiting since it now costs money):

```ts
export const POST = withAuth(
  withRateLimit(async (req) => {
    const { prompt } = await parseRequest(req, z.object({ prompt: z.string().min(1) }));
    const client = await ai.anthropic();
    const response = await client.complete({ messages: [{ role: "user", content: prompt }] });
    return Response.json({ text: response.text, usage: response.usage });
  }, { limit: 20, windowMs: 60_000, keyFor: (_req, { session }) => session.user.id })
);
```

The only differences between the free/local path and the paid/production path: which `ai.*()` factory is called, and whether `withRateLimit` wraps the handler. Validation, response shape, and usage tracking stay identical — that consistency is the whole design goal of `aiden-ai`.
